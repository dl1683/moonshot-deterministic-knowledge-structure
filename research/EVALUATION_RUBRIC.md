# Evaluation Rubric

This rubric defines pass/fail quality thresholds before implementation.

## Category A: Representation Quality

- A1: Paraphrase robustness
- A2: Distinction preservation for near-neighbor facts
- A3: Explicit relation semantics

### Current Criteria (Testable)

- A1 Pass:
At least 95% of curated paraphrase pairs produce identical `core_id`.
- A1 Fail:
Any systematic paraphrase class (voice swap, alias substitution, unit rewrite) has less than 90% convergence.

- A2 Pass:
At least 99% of near-neighbor fact pairs (same entities, different predicate/roles/polarity) produce different `core_id`.
- A2 Fail:
Any collision cluster caused by unknown-role wildcarding or omitted identity fields.

- A3 Pass:
All relation assertions satisfy typed signatures, and permutation tests show:
`contradicts(A,B) == contradicts(B,A)` plus stable IDs for reordered derivation premises.
- A3 Fail:
Any relation class where endpoint order or duplicate premises can change relation identity.

## Category B: Temporal and Provenance Integrity

- B1: Dual-time support (valid vs transaction time)
- B2: Revision history and supersession semantics
- B3: Provenance and evidence trace completeness
- B4: Confidence composition determinism

### Current Criteria (Testable)

- B1 Pass:
For every test case with backdated late-arrival or rollback,
`query_as_of(core, valid_at, tx_asof)` returns only revisions with visible assert transactions at `tx_asof`.
Historical replay must remain stable before and after rollback transaction time boundary.
For migrated revisions carrying `origin_tx_id`, visibility must start no earlier than migration admission tx.
For multi-hop migration cutover, effective projection winner must be deterministic by `(policy_seq, precedence_tx_id, projected_version_key)` without pre-cutover retroactive changes.
- B1 Fail:
Any query result includes a revision from non-visible transaction state (retroactive ghost fact),
or migrated revisions appear before migration admission boundary,
or cutover changes become visible before their governing migration/policy transactions.

- B2 Pass:
Supersession only occurs for exact `(core_id, valid_interval)` matches.
